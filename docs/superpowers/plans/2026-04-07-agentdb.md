# @durable-streams/coding-agents AgentDB Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build `agentdb` inside `@durable-streams/coding-agents` as a TanStack DB-backed read/write model for shared coding-agent sessions. The raw durable stream remains the source of truth. `agentdb` materializes base collections from stream envelopes, exposes query-derived session views, and provides optimistic actions for prompts and approvals that always await durable confirmation.

**Architecture:** Two layers:

- **Layer 1: base collections**
  - materialized directly from coding-agent stream envelopes
  - preserve structured truth and replay determinism
- **Layer 2: query-derived views**
  - built with TanStack DB live queries
  - use the `darix` pattern: seed collection + `toArray(...)` + `concat(...)` + `coalesce(...)`
  - expose UI-ready timeline/header/approval views

**Write model:** UI does not append user intents directly to the stream. It calls `agentdb` actions. Every action:

- performs an optimistic local mutation
- appends a durable envelope with a `txid`
- waits for durable confirmation with `awaitTxId(txid)`
- rolls back automatically on failure

The local repo currently does not re-export `createLiveQueryCollection`, `toArray`, `concat`, or `coalesce` from `@durable-streams/state`, even though `darix` does. Also add the required DB dependency to `packages/coding-agents/package.json`. Preferred direction: depend on `@durable-streams/state` for repo-consistent API surface, and use its re-exported TanStack DB helpers once added.

## Core constraints

These are the design rules `agentdb` must preserve:

- The durable stream is the source of truth. `agentdb` is a projection plus optimistic write layer.
- Base collections must be replay-deterministic from stream history alone.
- Raw structured message parts must be preserved. Query views may concatenate text, but ingest must not throw away structure.
- Only durable user-intent writes are optimistic. Agent output is never invented optimistically.
- All optimistic actions must await durable confirmation via `txid`. This is not optional.
- Timeline views must be query-derived, not handwritten reducer output.
- Synthetic approval provenance messages are durable first-class events and must survive replay/resume.
- Claude and Codex differences must be normalized into shared collections without flattening away provider-specific metadata needed for debugging or future UI.

---

## Why this shape

`StreamDB` is close, but not sufficient by itself. `StreamDB` assumes the stream already contains state changes that map directly to collection updates. Coding-agent streams are richer and messier:

- one envelope may update several collections
- one assistant message may arrive as many parts/deltas before completion
- one approval flow may create a request row, multiple response rows, and a synthetic durable provenance note

So `agentdb` should borrow the **API shape** from `StreamDB` and the **query shape** from `darix`, while keeping a custom materializer for coding-agent envelopes.

## Row identity and replay rules

- [ ] Use deterministic row ids derived from stream offset plus envelope-local index where possible.
- [ ] Keep provider ids as data fields, not as the only primary key.
- [ ] A replay of the same stream history must produce byte-for-byte equivalent collection contents for deterministic fields.
- [ ] Optimistic rows must reconcile onto their durable versions by `txid` rather than duplicating.

Recommended helpers:

- `makeEnvelopeRowId(offset, localIndex)`
- `participantKey(name, email)`
- `messageStableKey(...)`
- `toolCallStableKey(...)`

## Query layer

The query layer is where UI-friendly views are assembled.

Do not manually reduce entire streams in the UI. Follow the `darix` entity-query pattern instead:

- local seed collection per view
- `createLiveQueryCollection(...)`
- `toArray(...)` for child rows
- `concat(...)` for ordered text assembly
- `coalesce(...)` for fallback fields
- normalize the query output after read if needed

### Primary query factories

- [ ] `createAgentTimelineQuery(db, sessionId)`
- [ ] `createPendingApprovalsQuery(db, sessionId)`
- [ ] `createSessionHeaderQuery(db, sessionId)`
- [ ] `createParticipantSummaryQuery(db, sessionId)`
- [ ] `createToolActivityQuery(db, sessionId)`

### `createAgentTimelineQuery`

This is the main UI view. It should return an ordered array of timeline entries with already-concatenated display text plus structured metadata.

Suggested entry kinds:

- `user_message`
- `assistant_message`
- `thinking`
- `tool_call`
- `tool_result`
- `permission_request`
- `approval_response`
- `turn_complete`
- `session_event`

Suggested timeline entry fields:

- `id`
- `sessionId`
- `kind`
- `createdAt`
- `participant`
- `text`
- `message`
- `turn`
- `toolCall`
- `permissionRequest`
- `approvalResponse`
- `sessionEvent`

### Text concatenation

The UI should not reconstruct assistant text itself.

- [ ] Store raw text fragments in `message_parts`
- [ ] Use `concat(...)` in query layer to build final display text
- [ ] Preserve structured parts on the returned row for advanced renderers

For v1, queries should expose both:

- `text` — concatenated plain text for simple UIs
- `parts` — structured ordered parts for richer renderers

## Action layer

UI must write through `agentdb` actions only.

### Required actions

- [ ] `startSession(input)`
- [ ] `prompt(input)`
- [ ] `respond(input)`
- [ ] `cancel(input)`

### Shared action contract

Every action must:

- [ ] create a `txid`
- [ ] write optimistic rows into relevant local collections
- [ ] append durable stream envelope(s) carrying the same `txid`
- [ ] call `awaitTxId(txid)`
- [ ] reconcile optimistic rows to durable rows on success
- [ ] rollback optimistic rows on failure

### `startSession`

Purpose:

- create the local session shell immediately
- allow prompts to appear instantly even if bridge/agent startup lags

Suggested behavior:

- optimistic `sessions` row in `starting` state
- append durable session-start/user-intent envelope if required by the stream model
- wait for confirmation
- later bridge lifecycle events move it to `started`

### `prompt`

Suggested behavior:

- optimistic `messages` row for the user prompt
- optimistic `message_parts` row for the prompt body
- optimistic `turns` row in `queued` or `starting`
- append durable `user_message` envelope with `txid`
- wait for confirmation

### `respond`

Used for permission responses.

Suggested behavior:

- optimistic `approval_responses` row
- optimistic `permission_requests.status` update if this response is tentatively effective
- append durable `control_response` envelope with `txid`
- wait for confirmation

### `cancel`

Suggested behavior:

- optimistic cancellation marker in relevant turn or permission state
- append durable cancel/control envelope with `txid`
- wait for confirmation

## Acceptance criteria

- [ ] A coding-agent session can be fully rendered from `agentdb` query output instead of raw envelope iteration.
- [ ] User prompts and approval responses appear immediately in the UI via optimistic actions.
- [ ] Every optimistic write awaits durable confirmation and rolls back on failure.
- [ ] Assistant/user text is concatenated in query output from preserved structured parts.
- [ ] Approval provenance, duplicate responses, and restart/replay behavior stay correct after materialization.
- [ ] Claude and Codex histories both populate the same shared collection model.
- [ ] Replaying the same stream history twice produces identical collection state for deterministic fields.
